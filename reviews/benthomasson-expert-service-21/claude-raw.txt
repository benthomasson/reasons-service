REASONING: The `_ALLOWED_TEXT_EXPRS` allowlist is a good security measure against SQL injection through `text_expr`. However, **the SQLite LIKE-based search has no ranking** — results come back in insertion order, which could surface irrelevant results before relevant ones. The `fts_clause` for SQLite uses `lower(text_expr) LIKE :q{i}` with `%term%` patterns — this is a full table scan per term, which will be slow on large datasets. More concerning: the LIKE parameters use `f"%{term}%"` — if a term contains `%` or `_` (SQL wildcards), they won't be escaped, potentially matching more than intended. This is low-risk since terms come from `re.findall(r'\w+', ...)` which strips special chars, but it's worth noting.

### expert_service/rms/api.py
VERDICT: CONCERN
CORRECTNESS: QUESTIONABLE
SPEC_COMPLIANCE: N/A
ISSUE_COMPLIANCE: N/A
BELIEF_COMPLIANCE: N/A
TEST_COVERAGE: PARTIAL
INTEGRATION: WIRED
REASONING: The dispatch pattern (`if _is_sqlite(): ... else: ...`) is consistent but very repetitive — every function duplicates the import and dispatch. More substantively:

1. **`import_network` PG path**: Calls `api.add_node()` then potentially `api.retract_node()` then loops over justifications calling `api.add_justification()` — each of these is a separate DB round-trip. For large networks this will be very slow. The SQLite path using `store.save(network)` is a single bulk operation, creating asymmetric performance characteristics.

2. **`search_beliefs_fts` SQLite path**: The FTS5 MATCH query at line ~345 joins `nodes n JOIN nodes_fts f ON f.id = n.id WHERE nodes_fts MATCH ?` — the `WHERE` clause references `nodes_fts` (the table) instead of `f` (the alias). This works in SQLite FTS5 (content table queries use the table name directly in MATCH), but is easy to misread.

3. **`count_beliefs` / `count_nogoods`**: Use raw `sqlite3.connect()` instead of `reasons_lib` APIs, creating a direct dependency on the internal schema of `reasons_lib`. If `reasons_lib` changes its table schema, these will break silently.

4. **`search` function**: The `isinstance(result, dict)` guard handles the case where `reasons_lib.api.search(format="dict")` might return a string, but this suggests uncertainty about the API contract.

5. **`list_negative_candidates` SQLite path**: Loads ALL IN nodes into memory with `get_status()` then filters in Python. This is correct but won't scale.

Tests cover count_beliefs, count_nogoods, import_network, and search_beliefs_fts. The PostgreSQL paths in the new functions are untested.

## Summary

**Overall assessment**: This is a well-structured dual-backend implementation. The main architectural decision — dispatch at the `rms/api.py` layer rather than creating a full repository abstraction — keeps the change manageable but creates significant repetition. The f-string SQL construction in search.py is secured by the `_ALLOWED_TEXT_EXPRS` allowlist.

**Key concerns**:
1. **Dead code**: `_build_or_tsquery()` in loop.py is unused
2. **Duplicated logic**: Stop words and term extraction duplicated between `db/search.py` and `chat/loop.py`
3. **Orphaned project on import failure**: `import_reasons` endpoint doesn't roll back the project if `import_network` fails
4. **grep_content semantic change**: FTS search + Python filter differs from original SQL ILIKE behavior
5. **No PostgreSQL path tests** for the new helper functions in `rms/api.py`
